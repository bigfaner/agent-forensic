# 1.2: Create tool accessor functions + promote stats API

## Description

Two changes in different packages:

1. **Create `internal/parser/tools.go`** with 4 tool name accessor functions that replace hardcoded string comparisons throughout the codebase.

2. **Promote 4 private functions in `internal/stats/stats.go`** to public by capitalizing: `extractFilePath` → `ExtractFilePath`, `extractToolCommand` → `ExtractToolCommand`, `buildHookDetail` → `BuildHookDetail`, `parseHookMarker` → `ParseHookMarker`. Update internal callers within stats.go.

- `design/tech-design.md` Interface 2 (Tool Name Accessors) + Interface 3 (Stats Public API)
- `docs/conventions/tui-data-contracts.md` §1 (accessor functions), §4 (normalize in parser layer)
- `internal/stats/stats.go` — functions to promote: lines 107-139 (extractFilePath, extractToolCommand), lines 346-370 (buildHookDetail, parseHookMarker)
- `internal/model/calltree.go:217` — existing `isAgentTool()` (move to parser/tools.go)

- [ ] `internal/stats/stats.go` exports `ExtractFilePath`, `ExtractToolCommand`, `BuildHookDetail`, `ParseHookMarker`

## User Stories

No direct user story mapping — these are developer-quality tasks (P1-7, P1-8) verified by code-level success criteria.

## Hard Rules

- Accessor functions MUST accept multiple aliases per tool name (Claude Code JSONL format varies across versions)
- Stats promotion is rename-only — do NOT change function logic
- Existing tests in stats_test.go must continue to pass without modification

## Implementation Notes

For `IsAgentTool`: move the existing `isAgentTool` from `internal/model/calltree.go:217` to `internal/parser/tools.go`, capitalize, add `"SubAgent"` alias.

For `IsReadTool`: `name == "Read" || name == "Read1"` (check actual JSONL for variants).
For `IsEditTool`: `name == "Write" || name == "Edit"` (both modify files).
For `IsFileTool`: `IsReadTool(name) || IsEditTool(name)`.

For stats promotion: use Go's `gorename` or manual rename. Verify with `go build ./...`.
