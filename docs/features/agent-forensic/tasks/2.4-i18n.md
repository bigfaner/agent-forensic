# 2.4: i18n System

## Description

Implement the internationalization system with locale loading, key lookup, and runtime language switching. Supports Chinese (default) and English. All UI labels, status messages, and error messages are externalized into YAML locale files.

## Reference Files

- `design/tech-design.md` — Interface: i18n (T, SetLocale, CurrentLocale)
- `design/tech-design.md` — Dependencies (gopkg.in/yaml.v3)
- `prd/prd-spec.md` — i18n Requirements section
- `ui/ui-design.md` — Internationalization section (locale map coverage)

## Acceptance Criteria

- [ ] `internal/i18n/i18n.go` implements `T(key string) string`, `SetLocale(code string) error`, `CurrentLocale() string`
- [ ] `internal/i18n/locales/zh.yaml` with all Chinese translations (panel titles, status messages, error messages, key hints, diagnosis labels, dashboard headers)
- [ ] `internal/i18n/locales/en.yaml` with all English translations
- [ ] Unknown locale code returns error; unknown key falls back to key itself
- [ ] Default locale is `zh`
- [ ] Language switch is instant (no restart)
- [ ] Unit tests: locale loading, key lookup, fallback, switch
- [ ] Test coverage >= 80%

## User Stories

- i18n requirement: all UI labels, status messages, error messages must be translatable
- Supports `--lang en` CLI flag and `L` key runtime toggle

## Implementation Notes

- Load YAML files at init time or on first call (lazy load)
- Store translations as `map[string]map[string]string` (locale → key → value)
- Thread-safe for concurrent access (use `sync.RWMutex` or init-once pattern)
- Key naming convention: dot-separated paths (e.g., `panel.sessions.title`, `status.loading`, `error.dir_not_found`)
