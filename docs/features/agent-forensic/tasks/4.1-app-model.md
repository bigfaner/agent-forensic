# 4.1: App Model & Page Assembly

## Description

Implement the root Bubble Tea App Model that composes all sub-models (Sessions, CallTree, Detail, Dashboard, Diagnosis, StatusBar) into the final TUI application. Manages focus cycling, view switching (main/dashboard/diagnosis), session data flow between panels, real-time monitoring orchestration, and terminal resize handling.

This is the "Page Assembly" task: it wires all UI components together, manages the three views (main-tui, dashboard, diagnosis), and handles the data flow from parser/detector to UI models.

## Acceptance Criteria

- [ ] `internal/model/app.go` implements `AppModel` as root Bubble Tea model
- [ ] Three-panel layout: left Sessions (25%) + right CallTree (upper 67%) + bottom Detail (lower 33%) + StatusBar (1 line)
- [ ] Focus cycling: Sessions → CallTree → Detail → Sessions (Tab key)
- [ ] Direct-access: 1 focuses Sessions, 2 focuses CallTree
- [ ] View switching: s toggles Dashboard overlay, d opens Diagnosis modal
- [ ] Session selection flow: Sessions model emits event → AppModel loads call tree via parser → CallTree updates
- [ ] Call tree node selection → Detail panel loads node content
- [ ] Diagnosis modal: on Enter jump-back, auto-expands parent Turn and scrolls to node
- [ ] Real-time monitoring: watcher events → ParseIncremental → CallTree adds nodes
- [ ] Terminal resize: recalculate panel widths/heights, display warning if < 80x24
- [ ] q key quits from main view, closes modal from overlay views
- [ ] L key switches language (calls SetLocale, all models re-render)
- [ ] Integration tests: full session flow (load → navigate → detail → diagnosis → dashboard)
- [ ] Test coverage >= 85%

## Implementation Notes

- AppModel holds all sub-models as fields: sessions, callTree, detail, dashboard, diagnosis, statusBar
- `Update()` delegates to sub-models based on focus and active view
- Use `lipgloss.JoinHorizontal` and `lipgloss.JoinVertical` for layout composition
- Focus management: `activePanel` enum (sessions, calltree, detail)
- View state: `activeView` enum (main, dashboard, diagnosis)
- View rendering: `View()` switches between main layout, dashboard overlay, diagnosis modal
- Custom messages for inter-panel communication (SessionSelectedMsg, NodeSelectedMsg, etc.)
- Watcher lifecycle: start on init, stop on quit, pause/resume on monitoring toggle
