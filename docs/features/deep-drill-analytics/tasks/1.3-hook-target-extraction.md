# 1.3: Implement Hook Target Extraction

## Description

Implement Interface 3 from tech-design.md: Parse Hook trigger records to extract type and target command. Define the `HookDetail` data model. Enhance the existing `parseHookMarker` to also extract target information.

## Acceptance Criteria

- [ ] `HookDetail` struct defined with `HookType`, `Target`, `TurnIndex`, `FullID` fields
- [ ] `ParseHookWithTarget(text string) string` extracts "HookType::Target" or "HookType" (fallback)
- [ ] Regex: `(?i)(PreToolUse|PostToolUse)\s+hook\s+(?:for|result)\s*:?\s*(\w+)` for PreToolUse/PostToolUse
- [ ] Stop and user-prompt-submit-hook types: Target is empty, FullID = HookType only
- [ ] Target extraction failure: returns HookType without `::` suffix
- [ ] Unit tests: "PreToolUse hook for Bash" → "PreToolUse::Bash", "PostToolUse hook result: allowed" → "PostToolUse", no match → original text
- [ ] Coverage target: 90% for new code

## User Stories

- **Story 5** (partial): Provides the Hook type::target grouping for the Dashboard analysis panel

## Implementation Notes

- Existing `parseHookMarker` uses `strings.Contains` for type detection; enhance to also extract target
- Hook output format: `<hook-type> hook for <tool-name>` or `<hook-type> hook result: <text>`
- For PreToolUse/PostToolUse: extract the tool name after "for" or "result:"
- For Stop/user-prompt-submit-hook: no target extraction needed
- `ParseHookWithTarget` returns a string (the FullID), not a HookDetail. HookDetail construction (with TurnIndex) happens at the stats aggregation level.
