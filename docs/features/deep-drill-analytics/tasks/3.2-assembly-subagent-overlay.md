# 3.2: Page Assembly - SubAgent Analysis Overlay (UF-2)

## Description

Wire the SubAgent Full-Screen Overlay (built in 2.2) into the app model's message routing. Add `a` key binding in Call Tree context (only when SubAgent node selected), register the overlay in the main model's Update/View chain, and handle message passing.

## Reference Files

- `design/tech-design.md` — Integration 6: SubAgent Full-Screen Overlay
- `ui/ui-design.md` — Component: SubAgent Full-Screen Overlay (UF-2) Interactions
- `prd/prd-ui-functions.md` — Navigation Architecture: Primary Navigation entry #3
- `internal/model/app.go` (or equivalent main model) — target file for message routing

## Acceptance Criteria

- [ ] `a` key binding registered in Call Tree context, active only when cursor on SubAgent node
- [ ] `a` on non-SubAgent node is a no-op
- [ ] Overlay opens with SubAgent data from `SessionStats.SubAgents`
- [ ] `Esc` closes overlay, returns to Call Tree with cursor on original SubAgent node
- [ ] Overlay renders above existing content (existing content dims)
- [ ] App model delegates to `SubAgentOverlayModel.Update()` when overlay is active
- [ ] App model includes `SubAgentOverlayModel.View()` in rendering chain when active
- [ ] Window resize propagates to overlay

## User Stories

- **Story 2** (integration): Full overlay flow — press `a`, see three-section view, press Esc to return
- **Story 2** (key guard): `a` on non-SubAgent node does nothing

## Implementation Notes

- Target file: main app model file (where key routing and Update/View chain are managed)
- Route registration: add `SubAgentOverlayModel` to the app's model composition
- Key binding: in the Call Tree's `Update()`, check if selected entry is a SubAgent before opening overlay
- Message routing: when `SubAgentOverlayModel.IsActive()`, delegate `Update()` and include `View()` output
- New file `subagent_overlay.go` was created in 2.2; this task wires it into the app
